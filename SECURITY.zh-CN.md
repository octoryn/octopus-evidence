# 安全策略

## 报告漏洞

安全漏洞请**不要提交公开 issue**。

请通过 GitHub Security Advisories(仓库 Security 页的 "Report a vulnerability")
或邮件 **security@octopusos.ai** 私下报告。请包含描述、复现步骤与影响。我们力争在
数个工作日内确认。

## 范围说明

Evidence 是一个**密码学原语**:它产出规范哈希、每条证据的完整性哈希,以及一条
防篡改哈希链。按设计,以下几处与安全相关:

- **默认下是可*察觉*篡改,而非可*防止*篡改。** 不加密钥时,证据的 `id` 与
  `integrity` 哈希、以及链的链接哈希,只是对公开数据的普通 SHA-256。**哈希是公开的**,
  因此拥有证据或链存储写权限的攻击者可以编辑它*并*重新计算出一致的哈希 —— 于是该记录
  在被伪造的同时仍能通过校验。不加密钥的哈希能察觉*意外*或*无特权*的篡改,但察觉不了
  一个同时能改写哈希的、有动机的攻击者。
- **要抵抗伪造,请加密钥。** 向 `createEvidence` / `verifyEvidence` 传入
  `integritySecret`,向 `buildChain` / `nextLink` / `verifyChain` 传入 `secret`
  (经由 `verifyChain(chain, { secret })`)。这会把哈希切换为带密钥的
  **HMAC-SHA-256**。证据的完整性 HMAC 覆盖**整条**证据 —— `kind`、`subject`、
  `actor`、`content` 与 `provenance` —— 因此没有该密钥,攻击者即便拥有对存储的完整
  写权限,也无法伪造或改动**任何**字段,包括归属与来源(*谁 / 何处 / 何时*),而不
  仅仅是 payload。用与创建时*相同*的密钥校验。像对待任何签名密钥一样**保护并轮换它**;
  轮换只会为之后的哈希重新奠基,不会回溯性地重新校验旧记录。
- **链的尾部截断无法自我察觉。** 一条裸链(无论加不加密钥)只证明它自身是一个自洽的
  **前缀**:有效链的一个有效前缀本身也是一条有效链,因此丢弃最新的链接 —— 一次回滚 ——
  能不被察觉地通过校验。HMAC 密钥能阻止*伪造*新链接,但**不能**阻止截断。中间的编辑、
  删除、插入与重排*仍会*被抓到。要抓住截断,请把头哈希与长度带外记录下来,并向
  `verifyChain` 传入 `expectedHead` / `expectedLength`(或自行比对 `chainHead(chain)`):
  `verifyChain(chain, { expectedHead, expectedLength })`。
- **恶意输入被拒绝,而非静默损坏。** `stableStringify`(每个哈希之下的编码器)对任何
  JSON 无法忠实往返的内容会**抛出 `TypeError`**,而不是为其产出哈希:非有限数值
  (`NaN`、`Infinity`)、`undefined`、函数,以及环状结构。因此一次“成功”的哈希绝不
  掩盖静默的数据丢失。`verifyEvidence` 在内部吸收了该抛出:对无法规范化的/恶意的存储
  content,它返回 `false` 而非抛出,因此可以安全地在遍历不受信数据的批处理循环中调用。
  但*直接*(经由规范哈希辅助函数)对攻击者可控 JSON 做哈希的调用方,仍应为该抛出做好
  准备(例如将其当作被拒绝的输入),而非让它击穿某条请求路径。
- **无网络出站,零依赖。** Evidence 不做任何出站 I/O,且零运行时依赖(哈希用 Node
  内置的 `node:crypto`)。存储证据、传输它,以及静态保护存储与任何 HMAC 密钥,是运维方
  (与消费它的仓库)的职责 —— 本包只做计算与校验;它不持久化、也不传输任何东西。
